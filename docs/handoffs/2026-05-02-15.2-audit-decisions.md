## 8. Special-case `judge_policy=score` retentions (single_object that keeps score)

These cases are `single_object` but **keep** `judge_policy=score` because the
prompt requires conceptual ("does it look like X?") evaluation that L2
constraints can't capture:

| Case | Why score policy is right despite single_object |
|---|---|
| CV-AMB-001 "Create an apple" | Need judge to verify "looks like apple"; geometric constraints can't |
| CV-AMB-005 "Make something interesting with primitives" | Open-ended; judge evaluates "interesting" |

ADR-005 Q3 was about COMPOSITION cases needing judge for **spatial coherence**.
SINGLE_OBJECT cases CAN have judge for **concept recognition** ‚Äî they're
orthogonal use cases. No tension, no contradiction.

## 9. CV-CMP-003 ‚Äî the one judge_policy flip

CV-CMP-003 currently has `judge_policy: "audit_only"` ‚Äî a v3-design-stage
experimental policy that means "judge runs but its scores don't enter
the main report aggregates." This is an orphaned setting:

- It's the only case in the entire 80-fixture corpus using `audit_only`
- The mode was meant for measuring judge variance against a known-good
  baseline ‚Äî but we never built that calibration anchor
- For a COMPOSITION case (stacking two cubes ‚Äî clear spatial intent),
  `score` is the correct policy under ADR-005 strict mode

Action: flip `audit_only` ‚Üí `score`. The `JudgePolicy.AUDIT_ONLY` enum value
itself stays in `schema.py` (no schema change in #15.2) ‚Äî it remains
available for future calibration use, just not used in starter_v3.

## 10. Edge cases ‚Äî written here so reviewers don't have to re-derive

**CV-AMB-001 "Create an apple"** ‚Äî implementation may produce 1 mesh
(just sphere) or 2 (sphere + stem). Tagged `single_object` because the
**user concept is one thing**. ADR-005's enum is by mesh count, but in the
1-vs-2 ambiguity window, "single concept" wins for clarity. The
`judge_policy=score` already captures the "does it look apple-like" check.

**CV-SAF-001 / CV-SAF-002 ("delete all")** ‚Äî final scene has 0 mesh.
ADR-005 has no `EMPTY` enum value; we don't add one for two cases. Tagged
`single_object` because the **operational complexity is minimum** ‚Äî this
is the closest existing classification. Reviewer should not be confused:
the field describes intended output complexity, and "0" / "1" both fall
into the simplest bucket.

**CV-SAF-003 "Add a cube to empty scene"** ‚Äî 1 mesh output, classic
`single_object`.

**CV-SAF-005 "Remove all then add a sphere"** ‚Äî final state is 1 mesh.
The intermediate "all gone" state isn't part of the scene_complexity
calculation (which describes the final scene). `single_object`.

## 11. Cost projection ‚Äî revision

ADR-005 originally estimated `~25/80 cases at score` post-#15.2 and a
~3.5√ó cost rise. **The actual audit shows the existing 80 starter_v3 +
synthetic corpus has only 7 cases that warrant `score`** (CV-AMB-001,
CV-AMB-002, CV-AMB-004, CV-AMB-005, CV-CMP-003, CV-CMP-004, CV-CMP-005)
‚Äî up from 6 today (CV-CMP-003 flipping is the only change).

Revised cost impact:
- Before #15.2: ~$3 per 200-case run
- After #15.2: **~$3.20** per 200-case run (+1 case at score, marginal)
- After #15.3-#15.11 LLM-authored hard cases land (~150-250 more cases,
  most COMPOSITION/FULL_SCENE ‚Üí most at score): **then** the ~3.5√ó rise hits

The dramatic cost jump in ADR-005 was the right concern but the wrong
location ‚Äî it lands with hard-case authoring, not with the audit pass.
ADR-005 prediction should be updated retroactively or noted in #15.2
PR description.

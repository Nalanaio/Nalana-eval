## ADR-001 — Synthetic generator covers ~50–80 simple cases; LLM-assisted authoring covers the rest

**Date**: 2026-04-26
**Status**: Active
**Supersedes**: portion of GitHub issue *"generate 200-300 test cases using grok/deepseek"*
**Related task**: [#13](../) Test case authoring pipeline

### Context

V3.0's L2 benchmark targets 200–300 cases at launch, scaling to 1000+. The original plan was to use an external LLM (grok / deepseek / similar) to author all of them. While exploring that path we built a programmatic synthetic generator (`fixtures/synthetic/generate_cases.py`, commit `35c05ad`) that produces 50 cases deterministically from primitive × color × size combinatorics.

The question: does the synthetic generator replace the entire issue?

### Analysis

The generator's strengths and limits:

| Dimension | Synthetic generator | LLM-authored cases |
|---|---|---|
| Determinism / reproducibility | ✅ same input → same output forever | ❌ randomness even with seed |
| Constraint precision | ✅ bbox / RGBA pre-computed | ❌ drift between prompt and constraint |
| Coverage of simple cases | ✅ Object Creation, Materials & Shading | overkill |
| Coverage of complex cases | ❌ no Stylized, no Multi-Object, no Edit | ✅ where they shine |
| Coverage of `judge_policy: score` | ❌ all `skip` | ✅ required for L3 testing |
| Cost | zero | ~$0.01-$0.10 per case |

The generator is **constraint-first** (start from constraint templates, fill in prompts). Complex cases require **concept-first** authoring (start from a user scenario, derive constraints). Architecturally these aren't substitutable.

### Decision

Split the original issue into two parts:

- **50–80 simple cases** (Object Creation primitives, materials, parameterized primitives) → covered by synthetic generator. Done in commit `35c05ad`.
- **150–250 complex cases** (Stylized Modeling, Multi-Object Composition, Edit Operations, Complex Workflows, all `judge_policy: score`) → folded into expanded scope of task #13: "Test case authoring pipeline" (LLM-assisted draft → schema check → drift detection → honeypot injection → human spot-check → ingest into `fixtures/llm_authored_v3/`).

The original GitHub issue is closed with a comment pointing to commit `35c05ad` (for the simple-cases portion) and issue #13 (for the complex-cases portion).

### Consequences

- `fixtures/synthetic/` is the authoritative home for cheap, deterministic primitive/material cases.
- `fixtures/llm_authored_v3/` (to be created in #13) will hold complex cases.
- `fixtures/starter_v3/` keeps the 30 hand-authored seed cases.
- Avoid future "let's just LLM-generate the whole benchmark" requests by linking back to this ADR.

## ADR-002 — Eval outputs (artifacts/, db/, calibration/baseline_results/) covered by .gitignore

**Date**: 2026-04-27
**Status**: Active
**Verified**: `git check-ignore -v` + `git ls-files | grep`

### Context

A team member raised concern that eval-run outputs (per-run screenshots, scene-stats JSONs, accumulating CSV history, judge cache) could pollute git history if accidentally committed — these can grow to hundreds of MB across many runs.

### Decision

`.gitignore` lines 11–23 cover all eval outputs:

```
# Run artifacts (per-run output folders)
artifacts/

# Local database (CSV files, judge cache)
db/*.csv
db/*.sqlite
db/*.sqlite-journal
!db/.gitkeep

# Calibration baseline results (these are local-only)
calibration/baseline_results/
calibration/reference_images/*
!calibration/reference_images/.gitkeep
```

`db/.gitkeep` and `calibration/reference_images/.gitkeep` are intentionally not ignored so the directory structure exists in fresh clones.

### Verification

Verified on 2026-04-27:

```bash
git check-ignore -v artifacts/ db/runs.csv db/judge_cache.sqlite \
                    calibration/baseline_results/ calibration/reference_images/foo.png .env
# All hit the right .gitignore line.

git ls-files | grep -E "^(artifacts/|db/.*\.(csv|sqlite)|calibration/baseline_results/)"
# (empty) — no eval artifacts tracked in git.
```

### Consequences

- The team can run benchmarks freely on local machines without polluting git history.
- New eval-output directory types must be added to `.gitignore` proactively. The pattern: anything generated by `python -m nalana_eval.cli` belongs in `.gitignore`.
- This decision applies to the *output* of the system. *Input* artifacts (test cases, prompts, calibration handbook) remain tracked.

## ADR-003 — Mixed-concerns PRs forbidden

**Date**: 2026-04-29
**Status**: Active
**Cautionary case**: PR #21 (brian-test, 2026-04-28)

### Context

PR #21 bundled four independent changes in a single merge:

1. **Agentic retry-with-feedback loop** (a behavioral feature)
2. **`contracts.py` multi-key list-wrapper unwrap** (a parser bug fix)
3. **`openai_runner.py` conditional `response_format` for restricted models** (a runtime bug fix)
4. **`prompts/eval_default.md` step-kind documentation overhaul** (a prompt-engineering change with explicitly-tagged "校准集影响：需要重跑 baseline")

When post-merge benchmarks showed pass-rate climbing from 0% (pre-PR) to 100% (post-PR), the team initially attributed the lift to (1) the retry loop. Subsequent retrospective on `db/attempts.csv` (n=119) and on the `4-29runs/` aggregate (n=30 real-model retries) showed the loop's actual rescue rate was 7.5%, concentrated in a single run. The bulk of the lift came from (4) — the prompt overhaul taught models the correct NORMALIZED contract format that they had been guessing wrong before. PR #21's bundled-ness made this attribution invisible until manual archaeology.

This is a measurement failure, not a code-quality failure: each of the four changes is individually sound. The problem is that no one — including the author and reviewers — could tell which change drove which observed effect.

### Decision

Future PRs in this repository must touch **one concern at a time**.

- A "concern" = one feature, one bug fix, or one refactor. ADRs and adjacent docs that materially depend on the code change may live in the same PR (e.g., a new feature shipping with its ADR is one concern).
- **Bug fixes can be combined only if they share a root cause.** "These three things were all broken in the same module" is fine; "these three things were both broken at once" is not.
- The `_TEMPLATE.md` handoff doc's *Scope* section must explicitly enumerate *In* and *Out* — anything not in *In* belongs in a separate PR.

The current cleanup PR (PR-A) is permitted to bundle 13 commits because they share the root cause "post-PR-#21 cleanup, including the docs/process infrastructure that should have existed before PR #21 to prevent this recurrence." This ADR itself is one of those commits; it takes effect for *future* PRs from now on.

### Consequences

- Code archaeology becomes possible: each lift / regression / behavioral change has exactly one PR to blame or thank.
- PR review burden goes down (smaller PRs, narrower diffs).
- Authors must do more PR-creation overhead. Mitigation: provide the `_TEMPLATE.md` and (eventually) a `/handoff` skill.
- A PR that mixes a feature with several incidental fixes should be rejected at review time and resubmitted as separate PRs, even if it costs a day of rework.

## ADR-004 — Retry-with-feedback loop default OFF; insufficient data to validate

**Date**: 2026-04-29
**Status**: Active (provisional — re-evaluate after Task #13)
**Data sources**: `db/attempts.csv` (host-side, 4-28 runs) + `4-29runs/` aggregate (Docker-side runs)

### Context

PR #21 added a `retry-with-feedback` mechanism: when a `pass@k` attempt fails, the next attempt's prompt is augmented with a structured summary of the prior failure (failure class, executed steps, scene snapshot). The mechanism was implemented as **always-on** alongside the `pass@k` loop.

Two questions arose post-merge:
1. Does this break V3's "fair single-shot model comparison" semantics?
2. Does it actually help, and by how much?

Retrospective on real-model retries (mock excluded — mock returns hardcoded output regardless of prompt and so its 52 retries with 0 saves were non-informative):

| Source | Retry attempts | Saved by retry | Rescue rate |
|---|---|---|---|
| host `db/attempts.csv` | 10 | 0 | 0% |
| `4-29runs/` aggregate | 30 | 3 | 10% |
| **Combined** | **40** | **3** | **7.5%** |

All 3 saves came from a single run (`run_20260429_94b59e4e`, gpt-5.4). Other gpt-5.4 runs and all gpt-5.5 runs showed 0 retry saves. The lift is concentrated and unreliable rather than uniform.

Bundled into PR #21 alongside the retry loop were three other changes (see ADR-003) — the prompt overhaul in particular is the more probable explanation for the observed pass-rate lift.

### Decision

- **Default `--retry-with-feedback` to OFF.** Single-shot pass@k semantics are the V3 baseline; opt in explicitly to test or run with retry.
- **Skip retry on `failure_reason` starting with `"API error:"`.** Those are auth / parameter / endpoint failures that retry context cannot fix; they only burn API budget.
- **Exclude `mock` model from rescue-rate / variance metrics.** Mock is a smoke-test fixture, not a real LLM evaluation target.
- **`attempts.csv` gains `had_retry_context: bool` and `iterations_taken: int` columns** so post-hoc loop-vs-no-loop A/B analyses don't require re-running benchmarks.

### Re-evaluation gate

After Task #13 lands ≥30 hard cases (defined as: cases that gpt-4-class models fail attempt-0 on at ≥30% rate), re-run benchmarks with `--retry-with-feedback` ON vs OFF on the same case set. Decision rule:

- **lift ≥10pp** on hard-case pass@3 → flip default to ON, ship as ADR-005.
- **lift 5–10pp** → keep default OFF; consider redesigning the retry-context format (currently raw failure-reason + executed-steps + scene snapshot; could include explicit constraint texts and step-kind parameter docs).
- **lift <5pp** at n≥30 → consider deletion. Code path remains for ad-hoc opt-in but is no longer maintained.

### Consequences

- V3 `pass@k` numbers remain comparable across runs and across model versions. Brian's 4-29runs results showing 100% pass on 5 cases revert to "single-shot 100% on those 5 cases" rather than "retry-augmented 100%" — clearer story.
- Cost-per-case stays at single-shot baseline by default. Users who opt into retry know they're paying ~50% more on average for ~7.5% lift on current fixtures.
- The `--retry-with-feedback` flag is a discoverable opt-in; users who specifically want production-style "give the model a second chance" behavior can enable it in one flag.
- This ADR is provisional; expected to be revisited and either upgraded to permanent decision or replaced by a follow-up ADR within ~1 month of Task #13 landing.

## ADR-005 — Test case taxonomy: drop TaskLength, add SceneComplexity, evaluate spatial coherence via L3 judge

**Date**: 2026-04-30
**Status**: Active
**Supersedes**: implicit `Difficulty` semantics in `nalana_eval.schema.Difficulty` (kept deprecated one cycle; removal targeted at v3.2)
**Depends on**: `ian/judge-empty-scene-guard` PR merged before #15.2 audit (otherwise empty-scene hallucination contaminates the new judge-based metrics)
**Blocks**: #15.1 (schema field additions), #15.2 (existing fixture re-audit)

### Context

Two issues with the current `Difficulty: Short | Medium | Long` field surfaced during PR-A end-to-end verification (2026-04-29) and the Q3 prototype experiment (Task #22, 2026-04-30):

1. **`Difficulty` conflates three independent dimensions** — prompt verbosity, scene complexity, and judgment difficulty. The PR-A audit found CV-AMB-004 "Make a simple house" labeled `Long`, while the expected scene is cube + cone — by any reasonable measure that's an inexpensive task. The original authoring intent of `Difficulty` was "step count," which is the same mental model as v2 ground-truth-step-replication — explicitly abandoned in V3.

2. **Spatial coherence on compositional cases** (e.g., "Build a simple table") is currently **untestable**. Hard `mesh_object_count >= 2` constraints let pathological outputs (5 cubes piled at the origin) pass L2. Task #22 prototype confirmed the L3 judge **can** distinguish coherent vs incoherent compositions on non-empty scenes — given that the empty-scene hallucination guard ships first.

These motivate replacing the muddy `Difficulty` axis with cleaner axes and shifting spatial-coherence evaluation from L2 (constraints) to L3 (judge).

### Decision

#### 1. Drop `TaskLength` axis (proposal rejected at design time)

Initially proposed as a prompt-verbosity replacement for `Difficulty`. Histogram of all existing 80 prompts (shortest variant per case) showed 100% cluster in 3-9 word range — the axis would be a constant column with no signal. Word count is also a poor proxy for model burden in 8K+ context regimes. Axis dropped before landing.

#### 2. Add `SceneComplexity` axis (manually authored — Q4)

```python
class SceneComplexity(str, Enum):
    SINGLE_OBJECT = "single_object"   # 1 mesh
    MULTI_OBJECT  = "multi_object"    # 2–5 mesh, no spatial relationships required
    COMPOSITION   = "composition"     # 2+ mesh that the AUTHOR INTENDS to require spatial structure
                                      # (e.g., table = top above legs); enforcement is the L3 judge's
                                      # job, NOT a hard relative_positions constraint
    FULL_SCENE    = "full_scene"      # 5+ mesh + materials, full environment
```

`scene_complexity` is a **required field** on `TestCaseCard` and reflects **author intent**, NOT derived from constraint shape. Author intent and constraint shape can intentionally diverge — a stylized "futuristic table" might be tagged COMPOSITION with loose constraints to give the model creative latitude. This decoupling lets `tools/drift_check.py` (#15.4) catch the inconsistencies that would otherwise go silent if the field were auto-derived from constraints.

#### 3. Spatial coherence on COMPOSITION cases evaluated via L3 judge (Q3 — Task #22 data)

For COMPOSITION-tagged cases, **do not** add `hard_constraints.relative_positions` requiring fixed spatial relations. Hard constraints kill creative interpretations (a hammock-style chair, an art-installation inverted table). Instead:

- `judge_policy: "score"` (not `skip`)
- `style_intent.acceptable_styles` expanded to ≥3 creative options (e.g., `["realistic", "minimalist", "stylized", "futuristic", "geometric"]`)
- Trust the multimodal LLM judge to recognize "this is a table-like composition" vs "this is 5 cubes at the origin"

The schema's existing `relative_positions` field stays in place as **opt-in** for cases where geometric strictness is part of the test intent (e.g., a future "industrial CAD precision" category). Evaluator implementation of `relative_positions` is deferred until such cases land.

#### 4. Deprecate the existing `Difficulty` field (Q1)

```python
class TestCaseCard:
    scene_complexity:  SceneComplexity              # NEW, required
    difficulty:        Optional[Difficulty] = None  # DEPRECATED — removal in v3.2
```

Existing 80 fixtures keep their `difficulty` value during the deprecation cycle; new fixtures (#15.3+) do not populate it. CLI flag `--difficulty-dist` keeps working but emits a deprecation warning. Removal one cycle from now after all fixtures are migrated.

### Migration of existing 80 fixtures (executed in #15.2 — Q5 strict mode)

The audit performs all three actions in one commit per case-file batch:

1. Add `scene_complexity` field to every case based on manual judgment (~30 sec/case = ~40 min total for 80)
2. For every case tagged `scene_complexity: "composition"` or `"full_scene"`:
   - Flip `judge_policy: "skip"` → `"score"`
   - Expand `style_intent.acceptable_styles` to include 3+ creative options
3. Document each re-tag and policy-flip in the #15.2 PR description

Estimated impact:
- ~10–15 of existing 80 cases re-tagged COMPOSITION/FULL_SCENE
- Benchmark cost rises from ~$3 (current, mostly skip) to ~$10 per 200-case run (with ~25/80 at score)
- `docs/USAGE_GUIDE.md` cost-projection table updated as part of #15.2 PR

### Consequences

- **Schema** gains one field (`scene_complexity`), deprecates another (`difficulty`). Migration is graceful (one cycle, not breaking).
- **Authoring overhead**: +~5 sec per case for the new field. LLM-assisted authoring (#15.3) auto-fills `scene_complexity` from the draft.
- **Drift check (#15.4)** gets a new rule: warn when `scene_complexity` and `hard_constraints` shape disagree (e.g., tagged COMPOSITION but `mesh_object_count.minimum < 2`).
- **Reporting** gains "by scene complexity" group-by tables; "by difficulty" table marked deprecated.
- **Benchmark cost rises ~3.5×** for runs that include composition/full-scene cases. Documented and accepted.
- **L3 judge reliability now matters more** — the empty-scene hallucination guard PR is a hard prerequisite for the #15.2 strict-mode audit. Without it, the new judge-based metrics get systematically inflated.
- **ADR-004 retry-with-feedback re-evaluation gate (#15.12)** becomes more meaningful: hard composition cases produce real failures with real judge signals; the retry loop's value can finally be measured on data that isn't mock-dominated.
